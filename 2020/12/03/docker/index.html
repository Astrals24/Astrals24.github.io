docker常见命令

docker start ：启动一个已经创建过的容器

docker run ：创建一个新容器并运行一个命令

docker exec：在运行的容器中执行命令

docker search mysql：查询Mysql镜像

docker pull mysql ：拉取Mysql（默认最新版）

docker安装流程

1.yum包更新到最新

2.安装需要的软件包

yum-util 提供yum-config-manager功能，另外两个是devicemapper驱动依赖的

3.设置yum源（阿里仓库）

docker-mysql配置主从同步

1.准备2台服务器或者docker下创建2个mysql容器

主：master

从：slave

[注]如果用在docker下创建2个mysql容器的方式配置 注意主从的UUID一致问题

2.修改master中的配置文件my.cnf 并重启mysql容器（主库操作）

**[注1]先docker exec -it [容器] /bin/bash 进入mysql容器，其路径一般是/etc/mysql/my.cnf **

[注2]该配置为简易版

[注3]server-id不能重复

[注4]

Slave的配置与master类似，你同样需要重启slave的MySQL。如下：
log_bin = mysql-bin
server_id = 2
relay_log = mysql-relay-bin
log_slave_updates = 1
read_only = 1
server_id：是必须的，而且唯一。

log_bin：slave没有必要开启二进制日志bin_log，但是在一些情况下，必须设置，例如，如果slave为其它slave的master，必须设置bin_log。在这里，我们开启了二进制日志，而且显示的命名(默认名称为hostname，但是，如果hostname改变则会出现问题)。
relay_log：配置中继日志，log_slave_updates表示slave将复制事件写进自己的二进制日志(后面会看到它的用处)。
有些人开启了slave的二进制日志，却没有设置log_slave_updates，然后查看slave的数据是否改变，这是一种错误的配置。

read_only：尽量使用read_only，它防止改变数据(除了特殊的线程)。但是，read_only并是很实用，特别是那些需要在slave上创建表的应用。

设定全局锁“flush tables with read lock;”，对应的解锁模式命令为：“unlock tables;”

经过实际测试，对于MySQL单实例数据库和master库，如果需要设置为只读状态，需要进行如下操作和设置：
将MySQL设置为只读状态的命令：
# mysql -uroot -p
mysql> show global variables like “%read_only%”;
mysql> flush tables with read lock;
mysql> set global read_only=1;
mysql> show global variables like “%read_only%”;

将MySQL从只读设置为读写状态的命令：
mysql> unlock tables;
mysql> set global read_only=0;

对于需要保证master-slave主从同步的salve库，如果要设置为只读状态，需要执行的命令为：
mysql> set global read_only=1;

将salve库从只读状态变为读写状态，需要执行的命令是：
mysql> set global read_only=0;

限制super用户写入 使用 super-read-only=on

参考资料高性能mysql复制篇

5.查看主库状态（主库操作）

SHOW MASTER STATUS; 或者 SHOW MASTER STATUS/G

[注1]SHOW MASTER STATUS/G 后面没有冒号 不然会报错

[注2]请记住此时表中File和POS的值 并且不要对主库有任何的CRUD操作 不然该值会发生变化影响后续的操作

[注3]

mysql> show master status;
+——————+———-+————–+——————+——————-+
| File | Position | Binlog_Do_DB | Binlog_Ignore_DB | Executed_Gtid_Set |
+——————+———-+————–+——————+——————-+
| mysql-bin.000120 | 666 | | mysql | |
+——————+———-+————–+——————+——————-+
字段含义：
File : 从哪个日志文件开始推送日志文件
Position ： 从哪个位置开始推送日志
Binlog_Ignore_DB : 指定不需要同步的数据库

4.docker中mysql服务无法成功重启

​ 问题描述：在配置mysql主从的过程中 需要重启mysql服务 但是docker restart [容器ID]/[容器名] 之后 status 一直是restarting 状态 并且ports为null，

问题解决：

  1. 查看docker logs [容器ID]/[容器名]

  2. 发现是容器mysql 中 /etc/mysql/my.cnf 出现了问题 （我这里的原因是之前配置主从server-id=1 -前面多了个空格）

  3. 由于mysql容器并未成功启动 无法通过docker exec -it [容器ID]/[容器名] /bin/bash 方式进入mysql容器内部修改配置文件所以只能利用docker cp 把docker容器中的配置文件复制到主机中，在主机中修改完成后 再复制到docker容器中，命令如下：

    用法：docker cp [容器ID]:容器中配置文件路径 需要保存文件到docker中的路径

    例：docker cp [容器ID]：/etc/mysql/my.cnf /home/my.cnf [注] 把/etc/mysql/my.cnf文件 复制到 /home/my.cnf

    然后直接修改 /home/my.cnf 配置文件

    最后再copy回去

​ 例：docker cp /home/my.cnf [容器ID]：/etc/mysql/my.cnf

​ 重启docker容器即可 docker restart [容器ID]/[容器名]

5.docker-mysql配置主从Slave-IO 为NO

报错：Got fatal error 1236 from master when reading data from binary log: ‘binlog truncated in the middle of event; consider out of disk space on master; the first event ‘mysql-bin.000001’ at 98, the last event read from ‘./mysql-bin.000001’ at 125, the last byte read from ‘./mysql-bin.000001’ at 179.’

解决方法：

1.在master 中 输入

show master status;

里面的file 和 pos 需要和 Master_Log_File 和 Master_Log_Pos 保持一致

change master to Master_Log_File=’XXX’,Master_Log_Pos=XXX;

2.在slave中输入

stop slave；

CHANGE MASTER TO
MASTER_HOST=’119.45.181.128’,
master_port=3306,
MASTER_USER=’slave’,
MASTER_PASSWORD=’slave’,
MASTER_LOG_FILE=’XXX’,
MASTER_LOG_POS=XXX;

start slave；

show slave status/G;

Slave_SQL线程会重新的尝试工作

[注] 任何对主数据库的CRUD都可能会导致pos发生变化 以致于主从不一致 此时需要重新CHANGE MASTER TO 可以通过锁表方式先防止写入等主从配置完毕再解锁
